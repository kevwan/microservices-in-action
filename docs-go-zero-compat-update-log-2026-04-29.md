# go-zero 文档兼容性与优化修改记录（2026-04-29）

## 1. 评估范围

本次已对仓库中的文档进行了全量巡检，覆盖：

- 根目录所有 `*-cn.md`
- `README.md`
- `code-explanation/*.md`

评估目标：

- 检查是否存在与 go-zero 能力边界不一致的表述
- 检查是否存在“零配置/完全透明/无需任何额外工作”等过度承诺
- 补充前置条件、依赖条件与生产实践约束
- 优化少量示例代码中的行为细节

## 2. 修改原则

本次修改遵循以下原则：

1. 不改动文章结构主线，只修正高风险表述与易误导内容
2. “框架内置能力”与“外部依赖/业务自实现”明确区分
3. 对已有实践型内容尽量保持兼容，只做准确性增强
4. 保留原有中文表达风格，避免大规模重写

## 3. 已修改文件与改动摘要

### 3.1 能力边界与前提条件修正

- `service-discovery-cn.md`
  - 修正“支持 etcd/consul/k8s DNS 且完全透明”等过度表述
  - 明确 etcd 是核心注册发现方案，K8s 下可用 DNS Target/Endpoints
  - 将“零代码侵入”调整为“低代码侵入”

- `jwt-auth-cn.md`
  - 修正“无需任何认证逻辑”的描述
  - 明确仍需实现登录签发、刷新策略、权限控制
  - 总结中将“零侵入”调整为“接入成本低”

- `prometheus-monitoring-cn.md`
  - 修正“无需任何额外配置、开箱即用”绝对化描述
  - 明确需配置 `Prometheus` 端口并部署抓取端

- `tracing-cn.md`
  - 明确链路追踪需启用 `Telemetry` 配置后生效
  - 修正“无需改动/自动全开”这类绝对化措辞

- `config-management-cn.md`
  - 修正“与 etcd/Nacos 集成”为框架原生能力的暗示
  - 明确动态配置通常依赖外部 SDK 或自定义监听逻辑

- `grpc-interceptors-cn.md`
  - 修正将“负载均衡”归类为拦截器能力的表述
  - 调整为追踪、日志、熔断、指标、恢复等拦截器范畴

- `cron-job-cn.md`
  - 修正“go-zero 内置 cron 包”表述
  - 改为“go-zero 基础能力 + robfig/cron 组合”

- `retry-backoff-cn.md`
  - 删除不稳定的“zRPC 内置重试配置开关”结论型写法
  - 改为“gRPC service config 或自定义拦截器”两类实现路径

- `load-balancing-p2c-cn.md`
  - 明确 P2C 默认行为的前提：已配置服务发现或 Endpoints

- `tls-cn.md`
  - 明确 TLS 可快速接入，但 mTLS 仍需额外配置客户端证书校验与轮换策略

### 3.2 风险提示与措辞优化

- `rate-limit-cn.md`
  - 将“天然支持分布式限流”调整为“正确部署 Redis 与 key 规划后可实现”

- `cache-design-cn.md`
  - 明确缓存防护能力的生效范围与接入方式相关（例如 goctl 生成 Model）

- `graceful-shutdown-cn.md`
  - 明确生产零中断需要与 K8s `preStop`、宽限期、探针配合

- `log-desensitization-cn.md`
  - 去除“最近”这类时效性词汇，避免文档随时间失真

- `request-validation-cn.md`
  - 前言调整为“go-zero 基础能力 + validator/v10 扩展实践”，避免误解为单一路径

### 3.3 示例代码小修

- `health-check-cn.md`
  - 自定义就绪检查示例中改为根据状态动态返回 HTTP 状态码，避免固定写 `200` 的误导

## 4. 本次改动后的收益

1. 文档与 go-zero 实际能力边界更一致，减少误导
2. 新读者更容易识别“框架自动处理”与“仍需业务实现”的分界
3. 生产实践前置条件更清晰，降低落地偏差
4. 示例代码更贴近真实运行行为

## 5. 第二轮文风改造（方案优先、减少代码）

在兼容性修正之后，又对部分代码密度较高的文章进行了第二轮重写，目标是把文风统一到“问题背景 -> 方案判断 -> 设计原则 -> 落地建议 -> 少量关键片段”。

### 5.1 本轮重写的文章

- `pagination-cn.md`
  - 从“大段 `.api`/Go/SQL 实现”改为“偏移分页 vs 游标分页”的决策视角
  - 重点强调用户心智、数据规模、排序键与接口返回语义

- `retry-backoff-cn.md`
  - 从“拦截器实现代码”为主改为“错误分类、退避策略、幂等前提、流量治理”主线
  - 保留最少配置语义，弱化大段样例代码

- `rate-limit-cn.md`
  - 从“限流器 API 使用说明”改为“配额体系、限流维度、超限处理、治理闭环”主线
  - 更强调全局、接口、用户、租户等分层策略设计

- `service-discovery-cn.md`
  - 从“配置示例堆叠”改为“注册、发现、摘除”的治理模型
  - 强化 etcd、Kubernetes DNS、静态 Endpoints 的适用边界与排障思路

- `prometheus-monitoring-cn.md`
  - 从“指标、PromQL、告警样例堆叠”改为“看板、黄金信号、标签治理、告警设计”主线
  - 更强调指标体系如何服务排障和决策

- `jwt-auth-cn.md`
  - 从“签发/刷新代码演示”改为“认证、鉴权、生命周期、吊销、权限模型”主线
  - 明确 `@jwt` 是认证入口，不是完整安全体系

- `graceful-shutdown-cn.md`
  - 从“代码与 K8s 样例并列”改为“停机闭环、流量切换、超时预算、资源收口”主线
  - 更强调平台与应用协同，而不是单点 API 使用

### 5.2 文风改造后的统一特点

1. 减少长代码块和大段配置清单
2. 优先解释“为什么这样设计”而不是“代码怎么写”
3. 优先写方案边界、适用场景、常见失败点
4. 仅保留少量关键片段作为概念锚点

## 6. 本次改动后的综合收益

1. 文档与 go-zero 实际能力边界更一致，减少误导
2. 文章从“代码教程”更接近“架构思路与技术方案说明”
3. 新读者更容易先建立判断框架，再进入实现细节
4. 目录整体文风更统一，适合持续扩写后续主题

## 7. 后续可选优化建议

1. 统一术语词典
   - 建议统一使用：
     - “框架内置”
     - “框架支持”
     - “可集成”

2. 补充“前置条件”小节模板
   - 每篇在开头增加：依赖组件、最低版本、是否需额外中间件/配置

3. 增加“版本适配说明”
   - 对涉及行为差异的篇章（如监控、追踪、重试）增加版本脚注

4. 增加文档一致性检查脚本
   - 可在 CI 中检测高风险短语（如“无需任何配置”“完全透明”）并人工复核
