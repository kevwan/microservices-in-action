# go-zero 服务注册与发现：从配置连接到生产治理

## 前言

服务发现表面上是在解决"怎么找到下游地址"，但真正的问题远不止这个。

在微服务系统里，一个服务通常会经历这些变化：

- 实例动态扩缩容
- 机器或 Pod 随时上下线
- 同一服务在多环境并存
- 调用方需要自动拿到可用实例列表
- 故障实例要尽快从流量路径中移除

所以，服务发现的本质是：

> 用一套统一的注册、订阅和失效机制，把服务地址从业务代码里剥离出去。

本文重点讲方案和治理思路，代码仅保留必要配置片段。

## 1. 为什么不能继续写死地址

写死地址的问题不只是麻烦，而是会让系统在扩容和故障时变得非常脆弱。

典型问题包括：

1. 新实例上线要改配置
2. 故障实例下线后调用方还在访问旧地址
3. 多环境共享配置容易串流量
4. 业务代码被基础设施细节污染

因此，服务发现首先是一种**解耦机制**，其次才是一种调用便利性能力。

## 2. 服务发现的核心模型

无论使用 etcd、Kubernetes DNS 还是静态 Endpoints，底层都围绕三个动作展开：

1. **注册**：服务启动后把自身地址发布出去
2. **发现**：调用方订阅某个服务名对应的实例列表
3. **摘除**：服务异常或主动退出时，把失效地址移出列表

可以把它理解成下面这个模型：

```text
服务提供方  ->  向注册中心声明“我是谁、我在哪”
调用方      ->  根据服务名获取“现在谁可用”
负载均衡器  ->  在可用实例中选择一个节点
```

这里最重要的不是注册中心本身，而是**服务名成为稳定入口，实例地址成为可替换实现**。

## 3. 在 go-zero 里应该怎么选方案

go-zero 在 RPC 场景下最常见的服务发现方式主要有三类：

| 方案 | 适用场景 | 优点 | 注意点 |
| ---- | -------- | ---- | ------ |
| etcd | 通用生产环境 | 注册、发现、Watch 机制完整 | 需要维护注册中心 |
| Kubernetes DNS Target | 纯 K8s 场景 | 与平台集成自然 | 更依赖 K8s 网络与 DNS 语义 |
| 静态 Endpoints | 本地开发、测试、少量固定节点 | 最简单直接 | 不适合动态扩缩容 |

推荐选择顺序：

1. **本地开发**：静态 Endpoints
2. **标准生产环境**：etcd
3. **纯 Kubernetes 场景**：Kubernetes DNS Target 或 Headless Service

## 4. etcd 方案适合什么团队

etcd 方案最适合：

1. 服务规模持续增长的团队
2. 非纯 Kubernetes 部署环境
3. 需要统一注册、Watch、摘除语义的场景

它的价值不只是在于能存一组地址，而在于：

- 实例可自动注册
- 客户端可实时感知变更
- 故障实例会被摘除
- 服务名和实例生命周期解耦

最小配置通常类似这样：

```yaml
Name: order.rpc
ListenOn: 0.0.0.0:8080

Etcd:
  Hosts:
    - 127.0.0.1:2379
  Key: order.rpc
```

对调用方来说，真正关键的是：

- `Key` 一定稳定
- 多环境不要冲突
- 命名要有统一规范

## 5. Kubernetes DNS 更像平台内方案

如果服务完全运行在 Kubernetes 中，且希望最大程度使用平台原生能力，那么 DNS Target 往往更自然。

它适合：

1. 基础设施高度 K8s 化
2. 团队不想再维护独立注册中心
3. 服务名、Service、DNS 体系已经成熟

这种方式的优势是：

- 与平台一致
- 依赖更少
- 运维面更统一

但要注意：

1. DNS 方案不是 etcd 的逐项等价替代
2. 服务注册、摘除语义更多依赖 Kubernetes 本身
3. 对 Headless Service、Service 拓扑和探针状态要理解更清楚

## 6. 静态 Endpoints 不落后，只是适用面窄

静态 Endpoints 常被误解为"低级方案"，其实它在下面这些场景非常合适：

1. 本地联调
2. CI 集成测试
3. 小规模固定依赖
4. 灰度期间人工指定目标节点

它的核心价值是简单、可控、可复现。

问题在于它不适合承担生产环境里的动态变化。

## 7. 服务发现真正容易出问题的地方

### 7.1 服务名不稳定

如果服务名频繁变化，调用方就失去了一个稳定入口。服务名应该像 API 域名一样稳定，而不是跟发布批次或临时环境绑死。

### 7.2 多环境没有隔离策略

比如 `dev`、`staging`、`prod` 共用同一个注册中心却没有前缀区分，就很容易串流量。

推荐做法：

```text
prod/order.rpc
staging/order.rpc
dev/order.rpc
```

### 7.3 探针与注册摘除不同步

服务准备退出时，如果流量还没切走，但实例已经开始停机，调用方就会拿到一个"看似在线、实际上正在退出"的节点。

这也是为什么服务发现一定要和：

- 优雅停机
- 健康检查
- 注册摘除机制

一起设计。

### 7.4 注册了不可达地址

这是生产里最常见的问题之一：服务把容器内地址或错误网卡地址注册出去了，结果调用方可以发现实例，但无法连通。

所以部署方案必须明确：

1. 注册的是哪块网卡
2. 调用方从哪条网络路径访问
3. Service / Pod / Host 网络模型是什么

## 8. 服务发现和负载均衡是一体两面

服务发现只负责告诉客户端"有哪些实例"，但真正的调用效果还取决于"从这些实例里怎么选"。

在 go-zero 里，服务发现通常会与客户端负载均衡联动。也就是说：

1. 注册中心维护实例集合
2. 客户端实时刷新本地实例列表
3. 负载均衡策略在可用实例中做选择

因此，服务发现设计不能只看注册成功与否，还要看：

- 实例变更传播是否及时
- 慢节点是否会继续被选中
- 故障节点是否能快速摘除

## 9. 推荐的落地策略

### 9.1 开发环境

- 优先静态 Endpoints
- 降低本地依赖复杂度

### 9.2 测试与预发布环境

- 可以使用单节点 etcd
- 提前验证注册、发现、摘除链路

### 9.3 生产环境

- 标准场景优先 etcd 集群
- 纯 K8s 场景可优先 DNS Target
- 必须配合健康检查和优雅停机

## 10. 排障时应该看什么

服务发现问题排查建议按这个顺序：

1. 服务有没有成功注册
2. 注册的 `Key` 是否正确
3. 注册的地址是否可达
4. 客户端是否拿到了最新实例列表
5. 健康检查是否把异常实例及时摘掉
6. 是否存在环境串流量

如果只盯着代码本身，通常会漏掉真正的问题，因为这类故障很多时候发生在配置、网络和注册语义层。

## 11. 总结

服务发现不是一个"怎么连上别的服务"的小技巧，而是一项决定系统弹性和可运维性的基础能力。

设计时抓住这几个重点就够了：

1. 服务名必须稳定
2. 环境隔离必须清晰
3. 注册地址必须可达
4. 健康检查和优雅停机必须配套
5. 服务发现要和负载均衡一起看

一句话总结：

> 服务发现的目标不是让你少写地址，而是让实例可以自由变化，而调用入口保持稳定。
