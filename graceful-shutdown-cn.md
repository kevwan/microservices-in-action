# go-zero 优雅停机：从进程退出到流量切换闭环

## 前言

优雅停机并不是一个"收到 SIGTERM 后等几秒"的小技巧，而是一套覆盖**入口流量、在途请求、后台任务、外部连接**的退出策略。

如果只关心进程什么时候退出，而不关心流量什么时候切走、任务什么时候停止、连接什么时候释放，那么所谓优雅停机通常只是"慢一点的强杀"。

go-zero 提供了框架级停机能力，但要真正做到零中断发布，还必须把应用、注册发现、健康检查和 Kubernetes 编排放在一起看。

## 1. 优雅停机到底要解决什么

它主要解决四类问题：

1. 新流量继续打到即将退出的实例
2. 在途请求被强制中断
3. 消费者和后台任务半途退出
4. 数据库、MQ、第三方连接没有有序释放

所以优雅停机的目标不是"延迟退出"，而是：

> 先停止接新活，再处理完手头的活，最后安全离场。

## 2. 一个完整的停机闭环是什么样

成熟的停机流程通常包含这几个阶段：

1. 收到退出信号
2. 对外宣告自己不再可接收新流量
3. 停止接受新的连接或新的消费任务
4. 等待在途请求和处理中任务完成
5. 关闭外部依赖连接
6. 在超时边界内退出

如果缺少第 2 步，外部系统还会继续把流量送进来；如果缺少第 4 步，业务就会在处理中间被切断。

## 3. go-zero 在这条链路上负责什么

go-zero 更偏向负责**进程内的停机编排**，包括：

1. 监听停机信号
2. 停止 HTTP / RPC 服务接收新请求
3. 等待当前请求处理结束
4. 执行注册的 shutdown listener

这意味着 go-zero 能帮你把进程内部的停机流程做得更有序，但它并不自动替你完成所有平台协同动作。比如：

- Kubernetes 何时停止发流量
- 服务发现中的实例何时摘除
- 消息消费者如何完成当前消息后再停

这些仍然需要业务和部署侧共同设计。

## 4. HTTP、RPC、消费任务的停机侧重点不同

### 4.1 HTTP 服务

重点是：

- 不再接收新连接
- 等待当前请求处理完成
- 控制最大等待时间

### 4.2 gRPC 服务

重点是：

- 一元调用的收尾
- 流式调用的结束语义
- 长连接场景的断开节奏

### 4.3 MQ 或后台任务

重点是：

- 停止拉取新消息
- 等待已拉取任务完成
- 明确 ack / retry / requeue 语义

所以优雅停机不应被设计成一个统一的 `sleep`，而应根据工作负载类型分别处理。

## 5. Kubernetes 场景下最关键的是流量切换顺序

很多停机问题其实不是应用不会停，而是流量切换晚于进程退出。

推荐顺序是：

1. readiness 先失败
2. 平台停止向实例发送新流量
3. `preStop` 留出传播时间
4. 应用开始真正停机
5. 等待在途请求完成

如果顺序反过来，比如进程先进入停机但 readiness 还健康，就会出现经典问题：

- 旧实例已经在收尾
- 上游还在继续转发请求
- 用户间歇性收到 502、EOF 或超时

## 6. 超时预算要如何分配

优雅停机的超时并不是单独存在的，它要和平台宽限期共同组成一份预算。

可以按下面方式理解：

| 阶段 | 预算用途 |
| ---- | -------- |
| 流量切换时间 | 等待 readiness 失效传播 |
| 在途请求完成时间 | 给请求自然收尾 |
| 资源释放时间 | 关闭连接、刷写状态 |
| 最终兜底 | 防止无限等待 |

如果 `terminationGracePeriodSeconds` 只有 30 秒，但你把 `preStop`、请求收尾、资源释放都堆在这 30 秒里，又没有明确分配，最终通常会因为预算不够而被强杀。

## 7. 业务代码最应该补的不是停机代码，而是可中断性

很多停机失败，并不是框架没有 shutdown hook，而是业务任务本身不可中断。

典型问题：

1. 长循环不检查 context
2. 外部调用没有超时
3. goroutine 启动后没人收口
4. 消费任务没有 drain 机制

也就是说，优雅停机的前提是：

> 你的业务执行单元本身必须具备可停止、可等待、可收尾的特性。

## 8. 停机时哪些对象需要明确收口

建议逐项盘点：

1. HTTP server
2. gRPC server
3. 消息消费者
4. 定时任务调度器
5. 异步任务队列
6. 数据库连接池
7. Redis / MQ / 第三方客户端
8. 本地缓存刷写或状态持久化

如果其中任意一项在停机时仍继续接收任务或没有明确退出边界，就可能拖垮整个停机质量。

## 9. 一个可执行的停机顺序建议

在 go-zero 项目里，比较稳妥的顺序通常是：

1. 在 shutdown listener 中先标记实例不可用
2. 停止入口服务接受新请求
3. 停止消费者拉新任务
4. 等待在途请求与任务完成
5. 关闭外部连接
6. 到达超时后强制退出

其中最关键的是第 1 步和第 3 步，这两步做不好，就会不断有新工作流入系统。

## 10. 常见失败场景

### 10.1 发布时仍有零星 502

通常说明流量切换晚了，或 `preStop` 预留不够。

### 10.2 Pod 经常被 SIGKILL

通常说明在途任务收尾时间超过了平台宽限期，或者任务根本不可中断。

### 10.3 消费者停机后出现重复消费

通常说明消息 ack 时机和停机收尾顺序没设计好。

### 10.4 停机越来越慢

通常说明有 goroutine 泄漏、长尾请求、外部依赖阻塞或资源释放顺序不合理。

## 11. 推荐的最小落地清单

上线前至少确认这些问题：

| 检查项 | 是否建议必须具备 | 目的 |
| ---- | ---------------- | ---- |
| readiness 可主动失败 | 是 | 先切流量 |
| `preStop` 预留时间 | 是 | 等平台传播 |
| 请求处理支持 context 取消 | 是 | 避免长尾失控 |
| 消费者支持 drain | 是 | 避免半途退出 |
| 外部连接有关闭顺序 | 是 | 避免资源悬挂 |
| 停机日志与指标 | 推荐 | 便于排障 |

## 12. 总结

go-zero 能帮助你把进程内部停机流程组织起来，但真正的优雅停机是一个跨层协作问题，至少要同时满足：

1. 流量先切走
2. 新任务先停止进入
3. 在途任务允许收尾
4. 资源按顺序释放
5. 超时预算有清晰边界

一句话总结：

> 优雅停机不是“慢慢退出”，而是“先切流量、再清现场、最后退出”。
