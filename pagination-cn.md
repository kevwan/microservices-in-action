# go-zero 分页查询：从接口体验到数据层方案

## 前言

分页几乎存在于所有列表接口中，但它从来不是一个单纯的 `LIMIT/OFFSET` 问题。真正决定分页方案的，通常不是代码怎么写，而是下面这些问题：

- 用户是否需要"跳到第 N 页"
- 数据是否会频繁新增、删除、插队
- 列表是否会增长到百万级、千万级
- 前端更关心"翻页体验"，还是"总数准确性"

因此，分页设计本质上是一个**接口体验、数据库性能和一致性语义**之间的权衡问题。本文主要讲思路与技术方案，代码只保留必要片段。

## 1. 先决定目标，而不是先选语法

分页方案可以先按业务目标分为三类：

| 目标 | 更适合的方案 | 原因 |
| ---- | ------------- | ---- |
| 后台管理、报表、需要跳页 | 偏移分页 | 用户明确以页码思考 |
| 信息流、动态列表、移动端连续加载 | 游标分页 | 对深分页和数据抖动更友好 |
| 同时要跳页和高性能 | 混合方案 | 热区用偏移，深区用游标 |

一个很常见的误区是：

1. 先把分页写成 `page + pageSize`
2. 上线后才发现深页性能差
3. 再被迫引入游标分页重做接口

更稳妥的方式是，在接口设计阶段就问清楚：

- 这个列表是不是天然连续消费型
- 它未来会不会成为大表
- 总数是否真的必须强一致

## 2. 偏移分页适合什么场景

偏移分页的核心特征是：

- 请求参数简单：`page`、`pageSize`
- 用户心智清晰：第 3 页、第 10 页
- 前端容易做页码器和总数展示

它最适合：

1. 运营后台
2. 审计列表
3. 低频查询的管理页面
4. 总量不大，或只会访问前几页的列表

### 2.1 它的问题不在语法，而在数据规模

偏移分页的主要问题有两个：

1. 深分页代价高
2. 数据变动时容易出现漏读、重复

例如第 100000 页，本质上是先扫描前面大量数据，再丢弃大部分结果。这会带来：

- 扫描成本高
- 排序成本高
- 回表成本高
- 结果页越深，响应时间越不可控

### 2.2 什么时候还可以放心使用偏移分页

如果符合下面任意一项，偏移分页通常仍然是合理选择：

1. 数据量中等，主要访问前几十页
2. 查询条件固定，且有良好索引
3. 用户必须看到明确页码和总数
4. 后台场景对偶发慢页容忍度较高

### 2.3 偏移分页的优化重点

不是先想怎么写代码，而是先做这几件事：

1. 明确排序字段，避免无序分页
2. 用覆盖索引或延迟关联降低深页成本
3. 控制最大 `pageSize`
4. 对超深页做保护，避免无意义访问
5. 对总数统计做降级策略

一个常见的优化思路是先拿主键，再回表取完整数据：

```sql
SELECT o.*
FROM `order` o
INNER JOIN (
    SELECT id
    FROM `order`
    WHERE user_id = ?
    ORDER BY id DESC
    LIMIT ? OFFSET ?
) t ON o.id = t.id
ORDER BY o.id DESC;
```

这里真正重要的不是 SQL 长什么样，而是它背后的思路：

- 先让索引完成排序和截断
- 再缩小回表范围

## 3. 游标分页解决什么问题

游标分页不是为了"高级"，而是为了解决偏移分页不擅长的问题：

- 深分页性能差
- 数据变动时页内容不稳定
- 移动端连续下拉场景需要稳定续读

它的核心思想不是"跳到第几页"，而是：

> 从上一次看到的位置继续向后读。

所以游标分页更像是"读流"，而不是"翻页"。

### 3.1 游标设计的本质

游标通常应该满足三件事：

1. 能唯一标识一个位置
2. 能与排序规则严格一致
3. 对客户端尽量不暴露底层实现细节

因此最常见的游标组成不是单字段，而是：

- 时间戳 + 主键
- 分数 + 主键
- 业务排序字段 + 主键

这样做是为了处理排序字段重复的问题。只用 `created_at` 做游标，常常会出现同一毫秒多条数据无法稳定断点续读的情况。

### 3.2 游标分页最适合这些场景

1. 订单流、消息流、动态流
2. 评论列表、通知列表
3. App 首页、瀑布流
4. 一直向下翻、不需要精确跳页的场景

### 3.3 游标分页的代价

它也不是没有代价：

1. 不适合任意跳页
2. 调试难度高于偏移分页
3. API 契约更复杂
4. 前端需要处理 `nextCursor` / `prevCursor`

所以它不是替代所有分页，而是替代那些本来就不该用页码思维的列表。

## 4. 如何选排序键

分页稳定性的关键，不在分页参数，而在排序键。

一个合格的排序键应尽量满足：

1. 单调性较强
2. 可索引
3. 不易频繁变更
4. 能形成稳定顺序

推荐顺序：

1. `created_at desc, id desc`
2. `score desc, id desc`
3. `updated_at desc, id desc`（仅当业务确实以更新时间驱动列表）

不推荐：

1. 单独按 `status`
2. 单独按 `name`
3. 单独按非唯一时间字段

原因很简单：这些字段无法稳定定义"上一页最后一条之后是谁"。

## 5. API 设计应该怎么返回

### 5.1 偏移分页

偏移分页建议返回：

- `list`
- `page`
- `pageSize`
- `total`
- `hasMore`

适合后台和管理端。

### 5.2 游标分页

游标分页建议返回：

- `list`
- `nextCursor`
- `prevCursor`（如果需要反向翻页）
- `hasMore`

不建议同时强绑定 `page` 概念，否则接口语义会混乱。

### 5.3 `total` 要不要返回

很多列表接口并不需要强一致 `total`。总数统计往往是最贵的部分之一，尤其在复杂筛选下。

可以采用三种策略：

1. 强一致总数：后台报表场景
2. 延迟计算总数：先返回列表，再异步获取
3. 不返回总数：仅返回 `hasMore`

对于移动端信息流，第三种通常体验反而最好。

## 6. 数据层的落地原则

### 6.1 索引先于代码

分页查询的性能，70% 以上取决于索引设计，而不是逻辑层写得有多优雅。

偏移分页关注：

- 过滤条件
- 排序字段
- 联合索引顺序

游标分页关注：

- 排序字段是否可稳定比较
- 是否能命中联合索引

一个典型的游标分页查询形态：

```sql
SELECT *
FROM feed
WHERE user_id = ?
  AND (created_at < ? OR (created_at = ? AND id < ?))
ORDER BY created_at DESC, id DESC
LIMIT ?;
```

这类查询的重点是：

- 过滤条件和排序条件一起进入索引设计
- 用组合字段保证顺序稳定

### 6.2 限制参数边界

无论哪种分页，都建议加这些保护：

1. `pageSize` 上限
2. 最大可访问页深限制
3. 默认排序兜底
4. 非法游标保护

否则分页接口极易演变成性能放大器。

## 7. 常见错误设计

### 7.1 用偏移分页做无限滚动

问题：

- 新数据插入后，后续页会错位
- 用户连续下拉时容易看到重复项

### 7.2 游标只放一个时间字段

问题：

- 同时间戳多条记录时断点不稳定

### 7.3 列表排序和游标字段不一致

问题：

- 接口逻辑上能跑，结果却不可预测

### 7.4 总数统计默认强查

问题：

- 列表很快，总数反而成为瓶颈

## 8. 推荐决策表

| 业务类型 | 推荐方案 | 补充说明 |
| ---- | ---- | -------- |
| 运营后台列表 | 偏移分页 | 保留页码与总数 |
| 用户订单历史 | 偏移分页或混合 | 若总量很大，深页改游标 |
| 消息通知流 | 游标分页 | 强调稳定续读 |
| 评论流/动态流 | 游标分页 | 避免插队导致重复 |
| 导出任务列表 | 偏移分页 | 搭配总数统计 |
| 超大规模明细查询 | 混合 | 前几页偏移，后续游标 |

## 9. 总结

分页设计的核心不是"怎么写分页代码"，而是先回答三个问题：

1. 用户是按页使用，还是按流使用
2. 数据是否稳定，还是会频繁变化
3. 你更在意跳页能力，还是深分页性能

一句话概括：

- **偏移分页**是管理视角的分页
- **游标分页**是消费视角的分页

如果一个接口的本质是连续消费内容，就不要勉强它保留页码心智；如果一个接口本质是查第 N 页报表，就不要为了时髦强行上游标。
