# go-zero JWT 认证与鉴权：从 Token 校验到权限治理

## 前言

JWT 文章最容易写成一堆签发和解析代码，但在生产里，真正决定方案质量的不是"能不能发出 Token"，而是下面这些问题：

1. 认证和授权是否被清晰分层
2. Token 生命周期是否合理
3. 登出、吊销、刷新是否有闭环
4. 权限模型是否能随业务增长演进

go-zero 提供的是**JWT 校验入口能力**，也就是在 `.api` 中声明 `@jwt` 后，由框架帮你完成基础校验和 claims 注入。但登录签发、刷新策略、权限模型和风控约束仍然要由业务自己定义。

## 1. 先分清认证和鉴权

这两个概念经常被混用：

| 能力 | 回答的问题 | 典型落点 |
| ---- | ---------- | -------- |
| 认证 | 你是谁 | 登录、Token 校验 |
| 鉴权 | 你能做什么 | 角色、权限、资源访问控制 |

`@jwt` 解决的是前半部分的入口问题，但后半部分的权限控制，通常还需要配合业务中间件、角色模型或资源级校验一起完成。

## 2. JWT 适合什么，不适合什么

JWT 适合：

1. 无状态认证
2. 多实例横向扩展
3. API 网关或微服务统一透传身份

JWT 不适合单独解决：

1. 实时强制下线
2. 复杂会话管理
3. 细粒度资源权限决策
4. 高风险操作的二次确认

所以在架构上更合理的理解是：

> JWT 是认证载体，不是完整安全体系。

## 3. go-zero 中 JWT 该怎么接入

在 go-zero 中，推荐把 JWT 接入拆成三层：

1. **登录层**：验证账号密码，签发 Token
2. **框架层**：使用 `@jwt` 完成基础校验
3. **业务层**：基于 claims 做角色或资源权限判断

最小接入形态通常类似：

```api
service user-api {
    @handler Login
    post /api/user/login (LoginReq) returns (LoginResp)

    @jwt Auth
    @handler GetProfile
    get /api/user/profile returns (ProfileResp)
}
```

这一层的价值在于统一入口，而不是替代完整权限体系。

## 4. Token 设计的重点是生命周期

成熟的 JWT 方案通常不会只发一个长期有效 Token，而是区分：

1. **Access Token**：短期，用于访问接口
2. **Refresh Token**：长期，用于换取新的 Access Token

这样设计的原因是：

- Access Token 泄露后的风险窗口更短
- 用户不必频繁重新登录
- 服务端可以对刷新链路做更严格控制

建议思路：

| Token 类型 | 作用 | 建议时长 |
| ---- | ---- | -------- |
| Access Token | 接口访问 | 分钟级到小时级 |
| Refresh Token | 会话续期 | 天级到周级 |

时长不是越长越好，而是要和风险等级匹配。

## 5. Claims 里放什么最合适

Claims 设计的原则是：

1. 足够支撑认证和基础鉴权
2. 不放敏感明文
3. 不放高频变化的数据
4. 不把 Token 做成数据库快照

通常适合放：

- 用户 ID
- 角色标识
- 签发时间
- 过期时间
- 租户 ID（如果系统是多租户）

不建议直接放：

- 完整权限集合
- 用户详细资料
- 会频繁变化的业务状态

原因是 Token 一旦签发，短期内通常不可变，过多动态信息会很快失真。

## 6. 刷新、吊销和登出要形成闭环

JWT 最大的工程挑战，不是签发，而是如何处理"本来还没过期，但现在我想让它失效"。

常见方案：

### 6.1 短 Access Token + Refresh Token

优点：

- 简单
- 风险窗口可控

缺点：

- 无法立即让已签发 Token 全部失效

### 6.2 Redis 吊销名单

适合：

- 登出
- 风险账户强制下线
- 刷新令牌旋转

优点：

- 能补上"状态化撤销"能力

缺点：

- 需要额外存储与查询成本

更稳妥的生产方案通常是：

1. Access Token 短有效期
2. Refresh Token 旋转
3. 高风险场景配合 Redis 吊销名单

## 7. 权限模型不要停留在 `role == admin`

很多系统早期只用一个角色字段做权限判断，这在起步阶段可以接受，但随着业务演进，通常会很快失效。

建议按阶段演进：

1. **阶段一**：RBAC，角色控制接口访问
2. **阶段二**：角色 + 资源归属判断
3. **阶段三**：按租户、组织、资源类型做更细粒度控制

也就是说，JWT 里可以带角色，但真正的资源权限判断往往仍然要在业务逻辑里完成。

## 8. 安全设计应重点关注什么

### 8.1 Secret 管理

最基本的要求：

1. Secret 足够长
2. 不硬编码在仓库中
3. 用环境变量或密钥系统注入
4. Access 和 Refresh 使用不同 Secret 更稳妥

### 8.2 传输安全

JWT 一旦被窃取，攻击者往往就能直接冒用身份，所以必须默认运行在 HTTPS 之上。

### 8.3 算法与解析安全

不要把算法协商完全交给外部输入，签发与校验都应显式限定允许的签名算法。

### 8.4 风险接口的二次保护

即使已经通过 JWT，也不代表高风险操作就足够安全。支付、改密、提现等接口通常还应叠加：

- 二次验证
- 人机校验
- 风控评分
- 操作审计

## 9. 推荐的认证治理结构

一个更稳的 go-zero JWT 架构通常是：

```text
登录签发 -> @jwt 基础校验 -> claims 注入 -> 业务权限判断 -> 风控/审计补充
```

这里每一层职责不同：

- 登录签发解决身份建立
- `@jwt` 解决统一入口校验
- 权限判断解决“能不能做”
- 风控与审计解决高风险操作治理

## 10. 落地时最值得优先做的事

如果你的项目已经在用 go-zero JWT，但还没有形成完整方案，建议优先补下面这些：

1. 缩短 Access Token 生命周期
2. 引入 Refresh Token
3. 给登出和风险下线加吊销能力
4. 把权限判断从简单角色判断升级为资源级判断
5. 给登录接口叠加限流和审计

## 11. 总结

go-zero 在 JWT 场景里最有价值的地方，是提供了统一、低接入成本的认证入口；但真正的生产级认证方案，仍然要靠你把下面几件事补齐：

1. Token 生命周期
2. 刷新与吊销机制
3. 权限模型演进
4. Secret 与 HTTPS 安全
5. 风控和审计配套

一句话总结：

> `@jwt` 解决的是“怎么校验 Token”，而不是“怎么完成整套认证治理”。
