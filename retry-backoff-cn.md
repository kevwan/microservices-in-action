# go-zero 重试与退避：从可用性收益到重试风暴治理

## 前言

重试是分布式系统里最容易被滥用的策略之一。

做对了，它可以帮助系统跨过网络抖动、短暂故障和瞬时拥塞；做错了，它会把一个本来可恢复的小故障，放大成整条调用链的雪崩。

所以，重试设计的重点从来不是"怎么多发几次请求"，而是：

- 什么错误值得重试
- 多久以后再重试
- 重试几次后必须放弃
- 如何避免所有客户端同时重试
- 如何与幂等、熔断、限流配合

本文主要讲方案设计和决策思路，代码只保留关键落点。

## 1. 重试解决的是哪类问题

重试只适合处理**短暂性故障**，例如：

1. 网络瞬时抖动
2. 下游短暂不可用
3. 短时间资源竞争
4. 网关级限流后的可恢复退避

它不适合解决：

1. 参数错误
2. 权限错误
3. 业务状态错误
4. 明确不会因为时间推移而恢复的问题

一个简单判断标准是：

> 如果同一个请求稍后重放，成功概率会显著上升，它才可能值得重试。

## 2. 先区分错误，而不是先写循环

重试策略设计的第一步，是建立错误分类。

### 2.1 可重试错误

通常包括：

- `Unavailable`
- `DeadlineExceeded`（但要谨慎）
- `ResourceExhausted`
- 临时网络错误
- 上游明确标记为可重试的 429 / 503 / 504

### 2.2 不可重试错误

通常包括：

- 参数校验失败
- 未认证 / 无权限
- 资源不存在
- 业务状态冲突
- 幂等性无法保证的创建操作

如果业务层不先做这层分类，重试很容易变成一种"把失败延后几次再失败"的自我安慰。

## 3. 重试的五个核心参数

一套完整的重试策略，至少要回答下面五件事：

1. **最大尝试次数**：最多发几次
2. **初始等待时间**：第一次失败后等多久
3. **增长策略**：固定、线性还是指数增长
4. **最大等待上限**：防止等待时间无限放大
5. **抖动策略**：如何让多个客户端错峰

建议默认思路：

- 最大尝试次数：`3` 左右
- 退避策略：指数退避
- 抖动：必须开启
- 总耗时：不能突破上游请求 deadline

## 4. 为什么固定延迟通常不是好选择

固定延迟的问题不在简单，而在同步放大。

如果 1000 个客户端都在 1 秒后一起重试，那么它们并不是在帮助系统恢复，而是在**第二次同时打爆系统**。

对比三种常见策略：

```text
固定延迟：   1s -> 1s -> 1s
指数退避：   1s -> 2s -> 4s
指数+抖动：  0.7~1.3s -> 1.4~2.6s -> 2.8~5.2s
```

设计上几乎总是优先选择：

> 指数退避 + 抖动

原因是它同时解决了两件事：

- 失败越久，给系统越多恢复时间
- 多个客户端不再整齐地同时冲击下游

## 5. 什么时候不该重试

### 5.1 非幂等写操作

例如创建订单、发起扣款、发放优惠券，如果业务没有建立幂等保护，重试会直接制造重复副作用。

### 5.2 已经接近超时边界

如果请求整体 deadline 只剩几十毫秒，再做一次重试通常只会制造额外压力，而不会提升成功率。

### 5.3 下游已经明确进入熔断或过载状态

此时继续重试，价值通常小于伤害。

## 6. 重试必须和幂等性一起设计

重试策略本身并不保证安全，真正保证安全的是幂等约束。

对于写请求，建议先回答：

1. 请求有没有幂等键
2. 服务端是否能识别重复请求
3. 重复提交时是否返回同一结果
4. 下游链路是否也具备幂等能力

典型场景：

| 场景 | 是否建议重试 | 前提 |
| ---- | ------------- | ---- |
| 查询用户信息 | 是 | 天然幂等 |
| 更新资料 | 通常可重试 | 幂等覆盖写 |
| 创建订单 | 谨慎 | 必须有幂等键 |
| 发起支付 | 极谨慎 | 要有严格幂等与状态机 |

## 7. go-zero 里推荐把重试放在哪一层

在 go-zero 体系下，重试不要无差别散落在每个业务方法里，建议按职责分层：

### 7.1 RPC / HTTP 客户端层

适合放：

- 通用网络错误重试
- 通用退避策略
- 统一日志与指标

优点：

- 策略统一
- 便于观测
- 减少业务代码污染

### 7.2 业务编排层

适合放：

- 某些特定接口的重试开关
- 幂等性检查
- 与业务状态机结合的补偿逻辑

优点：

- 能理解业务语义
- 不会把所有错误都当成网络错误

推荐原则是：

> 通用重试放基础设施层，是否允许重试放业务层决策。

## 8. 重试要和哪些机制联动

### 8.1 与熔断联动

如果熔断器已经判断下游处于高风险状态，重试应当立即收敛，而不是继续放大流量。

### 8.2 与限流联动

当收到 429 时，不应立即重试，而应结合退避和配额策略处理。

### 8.3 与超时联动

重试是吃预算的。重试总耗时必须被包含在整个请求超时预算之内。

一个实用原则：

> 不要让下游重试预算超过上游总 deadline 的 30%~50%。

### 8.4 与日志和指标联动

至少要能观测：

- 每个接口的重试次数分布
- 最终成功率
- 重试后成功率
- 因重试带来的额外流量比例
- 因熔断/超时放弃重试的比例

没有这些指标，重试策略几乎无法持续优化。

## 9. 推荐策略矩阵

| 调用类型 | 建议策略 | 说明 |
| ---- | -------- | ---- |
| 读请求 RPC | 允许 2~3 次重试 | 指数退避 + 抖动 |
| 内部 HTTP 查询 | 允许短重试 | 必须受总超时约束 |
| 写请求 RPC | 仅在幂等前提下重试 | 否则风险高 |
| 第三方支付调用 | 谨慎 | 更适合状态查询 + 补偿 |
| 消息消费失败 | 不用同步重试堆栈 | 更适合重试队列 |

## 10. 在工程上如何落地

在实现层面，go-zero 项目通常有两条路径：

1. 使用 gRPC service config 定义基础重试行为
2. 使用自定义客户端拦截器统一实现退避与错误分类

如果团队希望策略统一、可观测、便于和业务规则联动，通常第二种更实用。

实现时最关键的不是代码量，而是下面这份配置语义：

```text
最大尝试次数: 3
初始退避: 100ms
最大退避: 5s
退避策略: exponential
抖动: enabled
可重试错误: unavailable, deadline_exceeded, resource_exhausted
```

## 11. 常见错误实践

1. 所有错误都重试
2. 没有抖动
3. 在业务代码里到处手写重试循环
4. 对非幂等写操作直接重试
5. 重试次数过多，导致请求尾延迟失控
6. 没有统一指标，重试是否有效完全靠猜

## 12. 总结

重试不是为了让请求"看起来更努力一点"，而是为了在**明确可恢复**的故障类型上，用受控的代价换取更高成功率。

真正成熟的重试策略应该同时满足：

1. 错误分类清晰
2. 退避节奏可控
3. 幂等前提明确
4. 和熔断、超时、限流联动
5. 有指标可观测

一句话总结：

> 重试不是循环问题，而是流量治理问题。
