# go-zero 限流机制详解：从配额设计到分布式治理

## 前言

限流不是一个"超过阈值就返回 429"的技术点，而是服务治理体系里最基础的流量整形能力。

真正有价值的限流设计，关注的是这些问题：

- 保护谁：用户、接口、租户，还是整个系统
- 限什么：QPS、并发数、带宽，还是业务操作次数
- 按什么维度切：全局、IP、用户、租户、接口、资源对象
- 超限后怎么办：拒绝、等待、降级，还是引导重试

go-zero 提供了基于 Redis 的限流能力，但真正决定效果的，往往不是调用哪个 API，而是**限流颗粒度和配额策略**本身。本文主要讲思路与技术方案，尽量少写代码。

## 1. 限流到底在解决什么问题

限流主要处理三类风险：

1. **资源保护**：防止单个接口或单个用户打满系统
2. **公平性**：防止少数高频调用方挤占公共容量
3. **业务约束**：例如登录尝试次数、短信发送频次、下单频率

也就是说，限流既可以是基础设施保护手段，也可以是业务规则的一部分。

## 2. 先确定限流维度，再谈算法

最常见的限流维度包括：

| 维度 | 适用场景 | 常见目标 |
| ---- | -------- | -------- |
| 全局限流 | 系统总容量保护 | 防止整体被打穿 |
| 接口级限流 | 热点接口保护 | 某单接口不会拖垮全站 |
| 用户级限流 | 公平性 | 防刷、防滥用 |
| IP 级限流 | 防攻击 | 防爬虫、防暴力破解 |
| 租户级限流 | 多租户 SaaS | 配额隔离 |
| 资源级限流 | 某个对象热点竞争 | 防止单商品、单活动被打爆 |

设计时最容易犯的错误是只做全局限流。全局限流虽然简单，但经常会把正常用户和异常流量一起误伤。

更合理的策略往往是多层组合：

1. 全局限流兜底
2. 接口级限流保护热点
3. 用户/IP/租户维度做公平性治理

## 3. 计数器和令牌桶该怎么选

go-zero 常见的两类限流思路分别适合不同场景。

### 3.1 计数器型限流

特点：

- 在一个时间窗口内限制固定请求数
- 规则直观
- 适合明确的业务配额

更适合：

1. 登录接口
2. 验证码发送
3. 订单创建频率控制
4. 管理后台接口配额

它的问题是边界突刺，也就是窗口边界附近会出现流量抖动。

### 3.2 令牌桶型限流

特点：

- 允许短时突发
- 长期平均速率受控
- 更适合流量整形

更适合：

1. API 网关
2. 突发型读请求
3. 需要平滑吞吐的公共服务

一句话选择：

- **规则型、业务型限制**：优先计数器
- **吞吐型、容量型限制**：优先令牌桶

## 4. 限流策略不是单一阈值，而是一张配额表

成熟系统里，限流通常不是一个固定数，而是一套按对象分层的配额体系。

例如：

| 对象 | 配额 |
| ---- | ---- |
| 免费用户 | 10 QPS |
| 付费用户 | 100 QPS |
| 企业租户 | 1000 QPS |
| 登录接口 | 每 IP 每分钟 5 次 |
| 发短信接口 | 每手机号每小时 10 次 |
| 秒杀商品 | 每商品每秒 500 次创建请求 |

这样做的价值在于：

1. 配额与商业模型一致
2. 风险和收益对齐
3. 热点对象有独立保护阈值

## 5. 分布式限流的核心不是 Redis，而是 key 设计

很多团队上来就讨论 Redis 脚本，但真正决定效果的通常是 key 设计是否合理。

常见 key 设计：

```text
global:api
api:/v1/orders
user:123:create_order
ip:10.0.0.1:login
tenant:acme:search
product:987:flashsale
```

一个好的限流 key 需要满足：

1. 能对应真实业务对象
2. 粒度不过粗，也不过细
3. 易于统计和排障
4. 不会造成无限 key 膨胀

例如用户级限流通常合理，但如果直接把完整 URL 查询串拼进 key，往往会制造大量无意义热点 key。

## 6. 限流放在哪一层最合理

### 6.1 网关层限流

适合：

- 全局流量整形
- 租户级配额
- IP 黑白名单
- 统一入口保护

优点：

- 在流量进入业务前就拦住
- 节省后端资源

### 6.2 服务层限流

适合：

- 业务动作频率控制
- 某个接口自己的容量保护
- 某类资源的热点保护

优点：

- 最了解业务语义
- 能按用户、订单、商品等对象精确治理

推荐做法通常不是二选一，而是组合：

1. 网关层做入口粗控
2. 服务层做业务细控

## 7. 限流后的处理策略也要提前设计

超限后并不是只有一种处理方式。

### 7.1 直接拒绝

最常见，也最简单。

适合：

- 非关键请求
- 明显异常流量
- 验证码、登录、防刷接口

### 7.2 短暂等待

适合：

- 内部系统
- 高价值请求
- 客户端可接受轻微排队

### 7.3 返回降级结果

适合：

- 推荐、搜索、聚合类接口
- 不一定要求强实时结果

### 7.4 引导客户端退避重试

适合：

- 平台型 API
- 有标准 SDK 的场景

所以限流策略应同时定义：

1. 阈值是多少
2. 超限后返回什么
3. 客户端是否要重试
4. 是否记录审计或告警

## 8. Redis 故障时怎么办

分布式限流通常依赖 Redis，这就带来一个工程问题：

> 如果 Redis 本身不可用，系统应该放行还是拒绝？

有两种策略：

### 8.1 fail-open

Redis 故障时放行请求。

优点：

- 限流组件不会成为单点
- 核心业务可继续提供服务

缺点：

- 流量保护能力临时失效

### 8.2 fail-close

Redis 故障时拒绝请求。

优点：

- 风险更可控

缺点：

- 限流组件故障会直接放大成业务故障

一般建议：

- 公共业务接口优先 `fail-open`
- 极高风险操作可以局部 `fail-close`

## 9. 限流和过载保护不是一回事

这两者很容易混淆，但职责不同：

| 机制 | 触发依据 | 目的 |
| ---- | -------- | ---- |
| 限流 | 预设阈值 | 控制流量规则和公平性 |
| 过载保护 | 实时资源状态 | 保护系统不被打崩 |

可以这样理解：

- 限流回答"本来就不该进来的流量是谁"
- 过载保护回答"现在即使该进来的流量我也扛不住了怎么办"

因此两者最好配合，而不是互相替代。

## 10. 推荐落地顺序

如果仓库或服务还没有系统化限流，建议按下面顺序演进：

1. 先加全局限流兜底
2. 给登录、验证码、短信等高风险接口加规则限流
3. 给热点读接口加令牌桶
4. 给多租户系统补配额体系
5. 接入指标和告警

不要一开始就追求极其复杂的多层脚本和动态配置，否则治理成本会先于收益到来。

## 11. 限流效果应该怎么观测

至少建议监控这些指标：

1. 请求总量
2. 超限拒绝量
3. 各维度命中率（用户、IP、租户、接口）
4. 限流后真实错误率是否下降
5. Redis 耗时和失败率

没有观测，限流就容易变成拍脑袋配置：

- 阈值太高，保护无效
- 阈值太低，误伤正常流量

## 12. 总结

限流设计的重点不是"会不会调用限流器"，而是这四件事：

1. 限流对象是谁
2. 配额按什么规则分配
3. 超限后如何处理
4. 如何与过载保护、熔断、重试配合

一句话总结：

> 限流不是一个算法选择题，而是一套流量治理规则。
