# go-zero 文章写作风格说明

## 目标

本仓库文章默认采用“思路与方案优先，代码为辅”的写法，重点帮助读者先建立技术判断框架，再进入实现细节。

## 推荐结构

每篇文章优先采用下面的结构组织内容：

1. 问题背景
2. 适用场景与边界
3. 方案对比
4. 设计原则
5. 落地建议
6. 常见误区或失败案例
7. 少量关键片段

## 写作原则

1. 优先回答“为什么这样设计”，再回答“如何实现”
2. 优先解释取舍、边界、前提条件，而不是堆砌 API
3. 减少长代码块、长 YAML、长命令列表
4. 保留的代码片段应只服务于解释关键概念
5. 避免把外部组件能力误写成 go-zero 原生能力

## 代码使用建议

1. 单篇文章尽量只保留 1 到 3 个关键代码块
2. 代码块长度尽量控制在能说明问题的最小范围
3. 非必须时，不展示完整项目骨架、完整 Handler/Logic/Model 三层样板
4. 对于配置示例，优先展示最小必要配置，而不是全量配置清单

## 推荐表达

建议优先使用这些措辞：

1. 框架内置
2. 框架支持
3. 可集成
4. 常见做法
5. 更适合
6. 不适合
7. 前提是

## 避免表达

尽量避免这些容易误导的说法：

1. 无需任何额外配置
2. 完全透明
3. 零侵入
4. 天然支持一切场景
5. 开箱即用即可覆盖生产复杂需求

## 判断标准

一篇文章在提交前，至少自查这几个问题：

1. 读者是否能先看懂问题和边界
2. 是否讲清楚了适用场景与不适用场景
3. 是否把 go-zero 能力和外部依赖区分清楚
4. 是否存在不必要的大段代码或配置堆叠
5. 是否能让读者在不看代码的情况下理解主要方案
