# go-zero Prometheus 监控：从指标采集到告警设计

## 前言

监控建设最常见的失败方式，不是工具没装上，而是把监控理解成"把指标暴露出来就结束了"。

真正有价值的监控体系，要回答的是：

1. 服务现在是否健康
2. 故障正在变严重还是正在恢复
3. 问题主要出在流量、延迟、错误还是容量
4. 哪一类业务正在受影响

go-zero 可以在配置好 `Prometheus` 入口后自动暴露一部分 HTTP 和 RPC 指标，但这只是起点。本文重点讲监控方案设计，而不是堆 PromQL 和样例代码。

## 1. 监控不是采集问题，而是决策问题

一个成熟的监控体系至少要覆盖三类目标：

| 目标 | 监控关注点 | 典型用途 |
| ---- | ---------- | -------- |
| 可用性 | 错误率、成功率、超时率 | 判断服务是否对用户可用 |
| 性能 | P95/P99、队列积压、吞吐 | 判断体验是否变差 |
| 容量 | CPU、内存、goroutine、连接数 | 判断是否接近资源边界 |

如果只盯着 QPS，往往会漏掉很多真正重要的问题。一个服务完全可能 QPS 很稳定，但延迟和错误率已经明显恶化。

## 2. go-zero 提供的监控能力适合放在哪一层

启用 Prometheus 指标后，go-zero 更适合作为**基础设施指标入口**，用于覆盖：

1. HTTP 请求耗时与状态码
2. RPC 调用耗时与状态码
3. Go runtime 与进程级指标

它适合回答：

- 哪个接口慢了
- 哪个接口错误多了
- 某个实例是不是资源异常

但它并不能自动回答这些业务问题：

- 今天支付成功率为什么下降
- 某个租户是不是被限流打满了
- 某类订单是不是卡在待处理状态

所以监控建设通常应分两层：

1. **框架指标**：看系统健康
2. **业务指标**：看业务结果

## 3. 先定义看板，再决定指标

很多团队是先把指标暴露出来，再临时拼 Grafana 面板。更稳妥的方式正好相反：

1. 先定义值班时要看的核心看板
2. 再反推每个看板需要哪些指标

推荐至少准备三类看板：

### 3.1 服务健康看板

关注：

- 请求量
- 成功率
- 错误率
- P95/P99
- 503/超时占比

### 3.2 实例资源看板

关注：

- CPU
- 内存
- goroutine
- FD 数量
- GC 压力

### 3.3 业务结果看板

关注：

- 下单成功率
- 支付成功率
- 待处理任务量
- 重试命中率
- 限流拒绝量

## 4. 黄金信号依然是最稳的起点

如果你不确定从哪里开始，优先围绕 Four Golden Signals 建监控：

| 信号 | 它回答的问题 | 常见落点 |
| ---- | ------------ | -------- |
| 延迟 | 服务是不是变慢了 | P95、P99、慢接口排名 |
| 流量 | 服务正在承受多少请求 | QPS、按路径分组 |
| 错误 | 用户是否正在失败 | 5xx、超时、业务失败率 |
| 饱和度 | 服务是否接近极限 | CPU、内存、goroutine、连接池 |

这套方法的优势是，即使你还没有很完整的业务指标，也能先把基础稳定性问题看清楚。

## 5. Prometheus 接入应关注什么

在 go-zero 中开启指标暴露通常只需要配置一个独立的 metrics 端口：

```yaml
Prometheus:
  Host: 0.0.0.0
  Port: 9101
  Path: /metrics
```

但真正影响可用性的，不是这段配置，而是以下设计点：

1. 抓取频率是否合理
2. 标签维度是否会爆炸
3. 是否区分环境、服务、实例
4. 指标是否能稳定聚合

例如路径标签如果直接带动态参数，就会把 Prometheus 变成高基数灾难源。

## 6. 指标设计最容易踩的坑是标签失控

下面这些标签通常是合理的：

- 服务名
- 方法名
- 状态码
- 区域
- 租户等级

下面这些标签要非常谨慎：

- 用户 ID
- 订单 ID
- 手机号
- 完整 URL
- 任意高基数字段

监控系统的原则不是"越细越好"，而是：

> 细到足够定位问题，但不能细到把监控系统本身打崩。

## 7. 告警设计不能只靠阈值

告警最常见的问题不是漏报，而是噪音过多，导致值班人员逐渐失去敏感度。

一个更实用的告警设计应该同时考虑：

1. **持续时间**：异常要持续多久才告警
2. **影响范围**：是单实例问题还是服务级问题
3. **告警等级**：warning 还是 critical
4. **恢复条件**：何时自动解除

推荐优先告警的对象：

1. 服务整体 5xx 错误率升高
2. P99 长时间恶化
3. 过载保护/限流拒绝量异常抬升
4. goroutine 或内存持续爬升
5. 某类核心业务成功率下降

## 8. 自定义业务指标该怎么补

业务指标不应该追求多，而应该追求能反映结果。

推荐优先补的指标：

1. 成功量
2. 失败量
3. 排队量/积压量
4. 外部依赖耗时
5. 核心状态分布

例如在订单系统里，比起采集几十个边缘指标，通常更应该先补：

- 订单创建成功总数
- 支付成功总额
- 待处理订单数量
- 第三方支付耗时分布

## 9. Kubernetes 场景下的重点不是会不会抓，而是能不能稳定抓

在 Kubernetes 中，常见做法是通过注解或 ServiceMonitor 让 Prometheus 自动发现目标。

重点关注：

1. Pod 重建后标签是否稳定
2. Service 和 Pod 之间的选择器是否准确
3. 抓取端口是否与业务端口隔离
4. 是否区分 `dev`、`staging`、`prod`

监控配置如果不做环境隔离，最终会得到一堆无法信任的混合数据。

## 10. 推荐的监控落地顺序

如果系统现在还没有形成完整监控体系，建议按下面顺序推进：

1. 先启用 go-zero 内置 Prometheus 指标
2. 建立基础服务健康看板
3. 配置最核心的服务级告警
4. 补充关键业务指标
5. 把 metrics 与 tracing、日志关联起来

不要一开始就做几十张大看板，否则维护成本会先把团队拖垮。

## 11. 一份最小可用监控清单

可以先检查服务是否已经具备下面这些能力：

| 项目 | 是否建议优先具备 | 目的 |
| ---- | ---------------- | ---- |
| 请求量与错误率 | 是 | 快速判断是否故障 |
| P95/P99 延迟 | 是 | 快速判断体验是否退化 |
| runtime 与进程指标 | 是 | 判断资源是否异常 |
| 业务成功率 | 是 | 判断用户是否真正受影响 |
| 告警抑制与分级 | 是 | 降低噪音 |
| tracing 关联跳转 | 推荐 | 提升排障效率 |

## 12. 总结

go-zero 的 Prometheus 能力适合作为监控体系的基础入口，但真正决定可观测性质量的，不是你暴露了多少指标，而是：

1. 看板是不是围绕问题组织
2. 标签设计是否可持续
3. 告警是否能反映真实风险
4. 业务指标是否能补上结果视角

一句话总结：

> 监控建设不是把 `/metrics` 暴露出来，而是让故障能被尽早发现、被快速判断、被稳定定位。
